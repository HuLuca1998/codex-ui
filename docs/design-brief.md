# Session Viewer — Design Brief / Prompt Pack

> **交付对象**：Claude Design（AI 设计师）
> **核心任务**：**重新构建整个页面框架**，并在新框架上完善 UI 表现。
> **重要前提**：本文档不规定任何布局结构。**有没有侧栏、顶栏放不放标题、
> 输入区是浮岛还是嵌入、设置是 sheet 还是页面 —— 全部由你决定**。
> 我只交付"用户需要做什么"和"它应该是什么气质"。
>
> 请把当前可能存在的实现忘掉。从一张空白画布开始。

## 1. 任务

为一款 macOS 原生工作台 App **从零设计**一套全新的页面框架与界面。

请按 2027 年的 Apple 美学水准，先**提出框架方案**（可以多个方向供选择），
再在选定的框架上交付完整设计系统与高保真稿。

**这是一次重构，不是一次美化。**

---

## 2. 关于"框架"我有什么倾向（这是最重要的一节）

### 2.1 我对"紧凑"过敏

我**不喜欢**那种"工程师工具"式的密集布局 ——
左边一栏列表挤满信息、中间一栏塞满消息、右边一栏堆满指标、
底部一条工具栏贴着十个按钮。

那种布局**让人疲惫**。看一眼就想关掉。

**我要的是：呼吸感、留白、慷慨的空间、克制的元素数量。**
**宁愿一屏少一些东西，也要让眼睛有地方放。**

### 2.2 但我也不要"空荡"

不是 Notion 那种 90% 白屏 + 一行字的极简。
不是 Apple 官网 landing page 那种"展示型"的稀疏。

我要的是**"有所主张的留白"** —— 每一处空白都是设计师**特意留下来的**，
不是因为信息不够，而是因为**信息需要被呼吸包围**。

> 想象一座**美术馆**：每幅画都有它自己的墙，墙之间有足够的距离让你后退去看。
> 而不是一个**仓库**：所有东西塞满每寸架子。

### 2.3 我对"框架"的预期

请提出**新的框架范式**。可以是任何形态。比如：

- 不一定有"左侧栏" —— 也许列表是从屏幕边缘抽出来的一层
- 不一定有"顶栏" —— 也许标题就漂在内容上方
- 不一定有"底部工具栏" —— 也许操作集中在某个浮岛 / 命令面板 / 上下文菜单
- 不一定是"三栏式" —— 也许是单焦点 + 周边环绕的辐射结构
- 也许是**单一深度**的扁平 → 失败
- 也许是**多层 Z 轴**的舞台 → 我喜欢
- 也许是**焦点驱动**的：用户在做什么，那件事就占据中央，其他退后

**请大胆。**框架本身就是这次设计的主角之一。
如果你只是给现有 Web App 范式重新刷颜色，这次合作就失败了。

### 2.4 框架方案的产出方式

请先给我 **2-3 个候选框架方向**，每个用一句话 + 一张低保真示意图说明：
- 它的核心组织逻辑是什么
- 它解决了"紧凑"问题的什么部分
- 它会让哪类用户特别舒服 / 不舒服

我会和你一起选一个深入。**不要直接进入高保真**。

## 3. 产品本质（先理解灵魂，再画一根线）

这不是一个聊天工具，不是仪表盘，不是 IDE。

它更像：

> **一台未来的"AI 控制塔"** —— 用户坐在它面前，不是为了输入什么，
> 而是为了**看见**：看见模型在想什么、在做什么、在调用什么、在写什么。
> 屏幕上的内容是**别的 AI 正在产生的**，本应用只是一面"诚实的玻璃"。

这意味着：
- 界面本身要**克制到接近隐身**，但不能死气沉沉
- 主角永远是**流过去的内容**，UI 是承载它的容器
- 但容器自己必须**有质感、有温度、有一点点冷酷的科技感**
- 用户应该会想长时间盯着它看 —— 像盯着水族箱、星图、雷达屏幕

---

## 4. 用户与场景（一段画面感的描述）

凌晨两点。一个工程师把这个 App 拖到了第二块屏幕的右半边，全屏。
左边是他的编辑器，他在写代码。右边这个 App 正在显示几条 AI 会话在并行跑：
有的在读文件，有的刚刚开始写一个 patch，有的在思考一个错误。

他不需要操作什么。他只是**偶尔瞥一眼**。
当其中一条会话开始有新动作，他的余光应该能**感觉到**，但**不会被打扰**。
当他真的需要看时，他**俯身靠近**屏幕 —— 这时界面应该立刻"迎接他"，
信息组织足够清晰，让他在三秒内读到关键内容。

**这个 App 必须同时是"远处的氛围"和"近处的工具"。**

## 5. 设计哲学（这是写给 AI 的提示词主体）

### 5.1 总纲

> **现代、超现代、2027 macOS 原生、悬浮、强交互感、慷慨呼吸。**

这六个词如果只是名词，是空的。我把它们翻译成可执行的感官指令：

### 5.2 "现代"——意味着什么

- 不解释自己。**好设计不需要标签、不需要工具提示、不需要"欢迎"**
- 每一个像素都在为信息服务。**装饰是失败的代名词**
- 类型层级是一切。**人眼应该在 0.3 秒内分清主次**
- 色彩极度克制。**只有在传递语义时才出现颜色**

### 5.3 "超现代"——意味着什么

> 这是这份简报最重要的一节。

**"超现代"不是"更花哨"，是"更安静地超越当下"。**

它**不是**：
- 不是 Material You 的色块堆叠
- 不是 Web3 / Cyberpunk 的霓虹与故障艺术
- 不是 Glassmorphism 的过度透明
- 不是 Neumorphism 的拟物按钮
- 不是 Material / Ant / Bootstrap 这些 2024 之前的 Web UI 库的衍生品

**它是**：
- 一种**陌生人看一眼会问"这是什么"**的视觉语言
- 一种让人觉得"如果 Apple 自己做下一代，可能就是这个方向"的设计
- 不靠堆元素、堆颜色、堆效果，而是靠**质感**与**节奏**赢的设计

**怎么实现，是你的事。我只判断结果。**
请不要急着掏出"加玻璃 / 加阴影 / 加渐变"这些 2024 年的套路 —— 给 2027 年一个新答案。

### 5.4 "2027 macOS 原生"——意味着什么

- 它必须看起来**像 Apple 自己会做的产品**，而不是第三方应用
- 但它领先半步 —— 不是抄 Sequoia / Tahoe，是**揣测 Apple 下一步会去哪**
- 红绿灯三键、系统字体、原生窗口圆角 —— 这些是基底，不能违和
- 它和 Notes、Messages、Things 3、Linear 并排打开时，
  **不应该是"另一个风格"**，而是"同一个家族里更前卫的一位"

### 5.5 "悬浮"——意味着什么

> 不只是阴影。是一种**让用户感觉到深度的能力**。

屏幕**不是平的**。但**怎么不平**，由你设计。
可以是叠层、光感、模糊、视差、动效、留白节奏，或者**别的我想不到的方式**。

我只要求最终用户能感觉到：
- 有些东西离他更近，有些更远
- 这个界面有空间，不只是平面
- 当一个元素被聚焦，它"靠近"了用户；其他东西"退后"了

**不要急着用"分四层 + 加玻璃"这种现成方案 —— 那是 2024 年的答案。**
**给 2027 年一个新答案。**

### 5.6 "强交互感"——意味着什么

> 这是最容易做错的一节。

**"强交互感"不是"动效多"，是"一切都有回应"。**

- 鼠标进入一个可点击区域 —— **必须有反馈**，哪怕只是 60ms 内的微亮
- 按下 —— **必须感觉到"被按下去了"**，可能是极轻的 1px 下沉
- 松开 —— **必须有回弹**，spring 曲线，不是 linear
- 数据进入界面 —— **必须有"它从远处来"的感觉**，不是凭空出现
- 选中状态切换 —— **必须有"光从左到右扫过"的感觉**，不是背景突变
- 拖拽 —— **被拖物轻微上浮 + 阴影变深**
- 滚动 —— **滚动条只在滚动时出现**，但**滚动本身有阻尼感**

更进一步（**这一点是 2027 感的灵魂**）：

> **App 必须像活着一样呼吸。**

当后端正在接收新数据，界面应该有一种"心跳" ——
极淡的边缘发光、状态点的呼吸、某一处的极轻闪烁。
**不能打扰阅读，但能在余光里被感知**。

### 5.7 "慷慨呼吸"——意味着什么

> 这是新增的一条原则，请尤其遵守。

- **空白是设计**，不是"还没填东西的地方"
- 每个元素的四周都应有让眼睛休息的距离
- 元素之间的间距应该**大于**你的第一直觉。**再大一点**。
- 一屏的信息量应该比同类工具**少 30%**
- 滚动是被允许的，**不要为了"装得下"而压缩**
- 行高、字距、padding 都要**比常规网页设计宽**
- 如果你犹豫"这里要不要再加点什么" —— **不加**
- 当用户视线扫过整个界面，**不应该有任何一处让眼睛卡住**

**慷慨呼吸 ≠ 空荡。区别是**：
- 空荡是"没东西"
- 慷慨呼吸是"有东西，但每样东西都有自己的领地"

### 6.1 视觉气质提示词

```
- 像 2027 年才会出现的 App
- 安静地超越当下，而不是更花哨
- 克制到接近隐身，但有温度
- 冷酷的科技感，但不冰冷
- 第一眼安静，越看越显眼
- 不迎合任何潮流，但绝不显得过时
- 让人说"这是什么 App"，而不是"这是 XX 框架做的"
- 像凌晨两点静静工作的设备
- 美术馆，不是仓库
- 用户三个月后仍不舍得换皮肤
- 你说不出它是哪一年的 —— 但绝不是过去
- 它不试图取悦你，但你忍不住靠近
```

### 6.2 空间与呼吸提示词

```
- 空白是设计，不是"还没填东西的地方"
- 间距比第一直觉再大一点
- 一屏信息量比同类工具少 30%
- 慷慨而不空荡 —— 每样东西都有自己的领地
- 行高、字距、padding 都比常规网页设计宽
- 滚动是被允许的，不要为了"装得下"而压缩
- 元素之间需要"看不见的引力场"
- 信息周围的留白本身就是一种强调
- 当用户视线扫过整个界面，不应有任何一处让眼睛卡住
- 宁愿一屏少几样东西，也要让眼睛有地方放
```

### 6.3 光与材质提示词（**仅当你的方案使用光与材质表达时适用**）

> 如果你的设计语言根本不靠光与材质 —— 跳过这一节，用你自己的语言。

```
- 光看起来像从屏幕内部来，不是反射进来
- 阴影是多层柔和叠加，不是单层硬投影
- 任何"玻璃感"都要有自己的密度，不是单纯透明
- 描边是半透白，不是纯灰
- 焦点是"被点亮"，不是"被框起来"
- 凹陷的内影是冷色，不是黑色
- 表面有极细的明度梯度，不是死板纯色
```

### 6.4 焦点与层次提示词

```
- 重要的内容靠近用户，环境信息退到后方
- 不要试图把所有功能都摆在第一层
- 把工具藏起来，让内容露出来
- 同时存在的事物，要有清晰的主次
- 当一个区域被聚焦，周围应有"让位"或"淡出"的暗示
- 不要用边框作为分隔的主要手段
- 当用户切换关注点，整个界面应"重新组织自己"
- 任何瞬间，屏幕上只有一个真正的主角
```

### 6.5 运动与生命感提示词

```
- 一切都有回应 —— 没有静态的可交互元素
- spring 曲线优于 ease 优于 linear；linear 永远禁止
- 入场动画有方向 —— 数据从下方升起，模态从中心展开
- App 必须像活着一样呼吸 —— 接收数据时有心跳
- 心跳不能打扰阅读，但能在余光被感知
- 鼠标悬停有磁吸感，元素轻微上浮 1-2px
- 按下有 1px 下沉，松开有回弹
- 选中状态的切换是"光扫过去"，不是背景突变
- 实时数据流入时有极轻的 glow pulse，<500ms 消散
- 滚动有阻尼感，不是即刻停止
- 加载状态永远不用 spinner —— 用呼吸、shimmer、或极细进度条
- 错误状态不用抖动 —— 用极淡的暖色描边浮现
```

### 6.6 微观细节提示词

```
- 滚动条只在滚动时出现
- focus ring 是双层的：强调色 + 半透晕
- 数字永远用 tabular-nums，让计数变化时不会抖动
- 时间戳是次要信息，不能比内容更显眼
- 等宽字体只用于 ID / 命令 / 代码 / 数字
- 图标是线性的、1.5px 描边，与 SF Symbols 视觉一致
- 不使用 emoji 当 UI 图标
- 圆角不是几何圆，是 squircle，看起来"刚停下来"
- 描边宽度永远是亚像素或 1px，绝不更粗
- 不使用粗体来强调 —— 用对比度和留白来强调
```

### 6.7 排版与节奏提示词

```
- 类型层级在 0.3 秒内可辨认
- 字号差距要"看得见"，至少 1.5pt 跳跃
- 中英文混排时，中文字距比英文略松
- 一行正文的字数控制在 50-70 中文字 / 70-90 英文字符
- 段落之间的间距应大于行距的 1.5 倍
- 引用、代码、命令各有自己的字体语言
- 副信息使用更小字号 + 更淡颜色 + tabular-nums
```

### 6.8 色彩与情绪提示词

```
- 深色底不是纯黑，是带冷调的深空
- 中性色阶至少 9 级，从底到顶有可见的明度梯度
- 强调色只有一个，绝不同时出现两个抢戏的颜色
- 语义色各有领地，互不越界
- 颜色的饱和度要低 —— 像被空气稀释过
- 颜色出现的位置就是"重要的位置" —— 颜色即信号
- 大面积不出现纯色 —— 只在小区域、徽章、状态点、描边
- 文字颜色至少四级：主标题 / 正文 / 副信息 / 占位
- 暗色模式下文字对比 ≥ WCAG AA，弱信息允许 AA-
```

## 8. 完整功能清单（详尽列举，布局由你决定）

> 这一节是**功能性需求**的完整规格。
> 每条都描述"应用必须支持什么"，**不描述"长什么样"或"放在哪"**。
> **容器、入口、组合方式、布局结构 —— 全部由你决定**。
>
> 但每一条功能都必须在你的设计中**有归宿** —— 不能漏。

---

### 8.1 数据来源与基本概念

应用要监控两个外部 AI 工具的本地会话日志目录：

- **Codex** —— OpenAI 的命令行 AI 编程工具，会话存储在 `~/.codex/sessions/`
- **Claude Code** —— Anthropic 的命令行 AI 编程工具，会话存储在 `~/.claude/projects/`

两个目录可能：同时存在 / 只有一个存在 / 都不存在。应用要优雅处理三种情况。

应用是**只读观测者** —— 它不创建会话内容（除了"新任务"功能），只是**解析并展示**那些 AI 工具产生的日志。

实时性由后端通过 **SSE** 主动推送（不是前端轮询）。
应用启动后保持长连接，事件秒级到达前端。

---

### 8.2 会话列表

需要让用户看到所有会话。每条会话应携带以下信息：

- **来源**：Codex / Claude（用某种视觉信号区分）
- **状态**：进行中 / 已结束 / 刚结束（最近 N 分钟有活动）
- **标题**：由 AI 自动生成的语义标题（如果有），否则用第一句用户消息
- **最新一句预览**：最后一条消息的开头（单行截断）
- **最后活跃时间**：相对时间（"3 分钟前"、"今天 14:32"、"昨天"、"周一"）
- **session id**：UUID 形式，应可缩写显示但需可见
- **工作目录**：会话运行时的 cwd，可缩写
- **token 累计用量**：可选显示
- **所属项目**：Claude 会话有"项目"归属（Codex 没有）
- **置顶状态**：被用户标记的会话
- **未读 / 新动作标记**：用户上次看过后又产生了新事件
- **父-子关系**：子代理（subagent）属于父会话，要表达层级

#### 8.2.1 子代理（subagent）
- Claude 会话可派生出多条子任务会话
- 父会话与子代理形成树状层级
- 子代理本身也是独立会话，可单独选中查看
- 用户应能**折叠 / 展开**父会话下的子代理
- 父会话应能显示**子代理数量**

#### 8.2.2 排序规则
- **进行中**的会话永远排在最前
- 然后按**最后活跃时间倒序**
- **置顶**会话在各自分组内提到上方
- 用户**无法手动拖拽排序** —— 排序由应用智能决定

---

### 8.3 会话筛选

用户可叠加以下筛选条件：

- **按来源**：全部 / 仅 Codex / 仅 Claude（三选一）
- **按时间范围**：全部 / 今天 / 本周 / 本月（默认"今天"）
- **按项目**（仅在选中 Claude 来源时可用）：从下拉中选某个项目
- 多个条件同时生效；当某条件被设置时应有清晰可见的反馈

筛选状态应能**一键清空**回到默认。

---

### 8.4 会话搜索

提供搜索能力：

- 匹配范围：**标题** / **工作目录路径** / **session id**（部分匹配）
- 搜索是**实时**的：输入即过滤，不需要回车
- 需要"清除"操作（一键回到全部）
- 应能看到当前匹配结果的**数量**
- 搜索与 §8.3 的筛选条件可叠加

---

### 8.5 选中并查看一条会话

用户选中一条会话后：

- 进入"密集阅读"模式
- 显示完整的时间线（按时间正序，最早在上 / 最新在下）
- 同时显示该会话的**元信息**：标题、来源、状态、工作目录、token 用量、session id
- 在会话列表中，当前选中的会话应有清晰的**高亮**反馈
- 切换会话应有过渡感（不是粗暴闪屏）

---

### 8.6 时间线事件类型（详细规格）

时间线由一系列"事件"组成，每个事件有不同的**类型**与**视觉权重**。
**对话与回答是主角，其他事件是配角**。配角默认应折叠，必要时展开。

需要支持的事件类型：

#### 8.6.1 用户消息
- 内容：用户输入给 AI 的文本（可多行、可很长）
- 可能包含 Markdown 元素
- 主角，应清晰、有视觉权重

#### 8.6.2 AI 助手回复
- 内容：AI 返回的文本
- 必须支持**完整的 Markdown 渲染**：
  - 标题（h1-h4）
  - 列表（有序 / 无序 / 嵌套）
  - 加粗、斜体
  - 行内代码、代码块（含语言标识 + 复制按钮）
  - 引用
  - 链接（点击系统默认浏览器打开）
  - 表格（含表头 / 表体）
  - 分割线
- 主角，视觉权重最高

#### 8.6.3 AI 思考过程（reasoning）
- 内容：AI 模型的"内部思考"（如 "让我先看看这个文件……"）
- 可能很长（几千字）
- **默认折叠**，作为配角
- 用户可展开看完
- 视觉上应"次于"对话内容

#### 8.6.4 工具调用
- AI 调用了某个工具（如 Read / Write / Search / Fetch / Bash 等等，工具名可多达几十种）
- 应显示：
  - **工具名**
  - **调用参数**（一段 JSON 或键值对，可能很长）
  - **调用状态**：进行中 / 成功 / 失败
  - **调用结果**（成功时的返回内容；失败时的错误信息）
- **默认折叠成一行**，点击展开
- 不同工具可有不同图标 / 颜色提示（但不必每个都不同）

#### 8.6.5 命令执行（shell）
- AI 在 shell 中跑了一条命令（属于工具调用的特例，但太常见，单独处理）
- 应显示：
  - **命令本身**（等宽字体）
  - **stdout** 标准输出（可能很长，需可滚动）
  - **stderr** 标准错误（如果有，与 stdout 区分）
  - **退出码**
  - **执行时长**（可选）
- 长输出需有"折叠 / 截断 / 展开全部"机制

#### 8.6.6 代码补丁（patch / diff）
- AI 对一个或多个文件做了修改，以 diff 形式呈现
- 应支持：
  - **文件名头**（filename，每个文件单独一段）
  - **多个 hunk**（变更块，含 `@@ ... @@` 行号信息）
  - 每行有类型：
    - **context**（上下文，灰色，无前缀）
    - **add**（新增，绿色系，前缀 `+`）
    - **del**（删除，红色系，前缀 `-`）
  - 行号显示
- 长 diff 需可滚动
- "改了什么 / 删了什么"必须一目了然

#### 8.6.7 网络搜索
- AI 调用了网络搜索工具
- 应显示：**搜索词** + **多条结果**（每条含 标题 + URL + 摘要）
- 结果链接可点击打开

#### 8.6.8 网页抓取
- AI 抓取了某个 URL 的正文
- 应显示：**URL** + **抓取到的内容摘要**（可能很长，需折叠）

#### 8.6.9 图像生成
- AI 生成了图像（不常见但要支持）
- 应显示：**图像本身**（可点击放大） + **生成参数 / 提示词**

#### 8.6.10 AskUserQuestion（AI 反问用户）
- AI 主动向用户提问，提供候选选项
- 每个问题含：
  - **问题文本**
  - **候选选项**（每个有"标签" + "描述"）
  - **单选** 或 **多选** 模式
  - **"Other"**：用户可输入自定义文本作为回答
- 用户回答后，该问题应标记为"已回答"，所选选项应高亮
- 同一时刻可能出现一组（多个）问题

#### 8.6.11 系统事件
- 会话开始 / 结束 / 中断 / 重连 / 错误
- 应低调呈现，不抢戏

#### 8.6.12 审查结论（findings）
- 某些工作流会输出"审查发现"，是一组小条目
- 每条含：**标题** + **描述** + **位置**（文件路径 + 行号）
- 一组 findings 应聚合呈现

#### 8.6.13 时间分隔符
- 当相邻事件间隔较长（数分钟以上），可在时间线上插入"时间标识"
- 提示"时间过去了 X 分钟" / "下午 3 点" 等

#### 8.6.14 截断提示
- 当某段内容过长被截断，应有清晰提示："省略了 X 行 / 字"
- 用户可一键展开全部

#### 8.6.15 原始 JSON
- 每一个事件都对应一段原始 JSON
- 工程师用户偶尔需查看（审计 / 调试）
- 默认隐藏；通过某种次级操作可显示

---

### 8.7 时间线的整体行为

- 一条会话可能包含 **5000+ 事件**，必须流畅滚动
- 默认按时间**正序**展示（最早在上、最新在下）
- 用户可上下滚动查看历史
- 应提供"**跳到顶部 / 跳到底部**"的快捷方式
- 应提供"**在当前会话中查找**"功能 —— 输入关键词，定位到匹配的事件
- 应能**临时切换显示哪些事件类型**（详见 §8.8）
- 当数据流入但用户不在底部时，应给出"邀请回到底部"的提示而非强行滚动（详见 §8.9）

---

### 8.8 事件类型过滤（阅读时的临时过滤）

阅读一条会话时，用户可临时切换显示哪几类事件：

- 只显示**对话**（用户消息 + 助手回复）
- 加上**思考**（reasoning）
- 加上**工具**（工具调用 + 命令执行 + diff + 搜索等）
- 加上**系统**（系统事件 + 审查结论）
- 默认四类全开

切换是即时的，过滤后被隐藏的事件不应留下视觉空洞。

---

### 8.9 实时跟随

会话进行中时：

- 新事件自动追加到时间线末尾
- **如果用户已在底部** —— 自动滚动跟随
- **如果用户已滚到上方**（在回看历史） —— **不要强行滚动**，应弹出一个**"↓ N 条新消息"**的可点击提示，邀请他跳回底部
- 用户应能**手动开 / 关**"自动跟随"
- 新事件进入时应有"它从某处来"的微妙视觉提示（**不能打扰阅读**）

---

### 8.10 续聊（Claude 会话专属）

用户可在一条 Claude 会话上接着发新消息：

- 提供**文本输入区**：支持多行、内容自适应高度、有最大高度上限（超过即内部滚动）
- 提供**发送**：可通过快捷键（如 Enter / Cmd+Enter）或点击按钮发送
- 输入区无内容时应不抢戏
- 发送中应有"正在发送"状态指示
- 发送成功后输入区清空
- 发送失败应给出原因
- 发送出去的消息应立即在时间线上以"用户消息"形式追加

> **Codex 会话目前不支持续聊**（CLI 限制），这种会话上不应出现"发送"功能；
> 输入区可改为"在当前会话中查找"用途，或干脆隐藏。

---

### 8.11 在 iTerm2 中继续对话（仅 macOS 桌面）

针对 macOS 重度用户，提供一个操作：

- 启动一个新的 iTerm2 窗口
- 自动跑 `claude --resume <sid>` 或 `codex --resume <sid>`
- 让用户回到他熟悉的 CLI 工作流
- 此功能在移动端 / 浏览器形态下不应出现

---

### 8.12 token 用量（单会话）

每条会话累积的 token 用量需可见：

- **实时累加**（新事件产生时更新）
- 应区分**输入 token / 输出 token / 总计**（如果数据可得）
- 数字必须使用 **tabular-nums**（避免计数变化时抖动）

---

### 8.13 用量报表（跨会话汇总）

应用应提供一个独立于"会话列表"的"汇总报表"视图：

- **按天**的 token 用量（柱图 / 时间线 / 数字卡）
- **按 AI 模型**（如 claude-opus / claude-sonnet / gpt-5 等）的用量分布
- **按项目**（Claude 项目维度）的用量分布
- **按会话**的用量排行
- **时间范围**可切换（今天 / 本周 / 本月 / 全部）
- 数字必须 tabular-nums
- 图表应朴素 —— 数字 + 微缩柱图，**不要花哨的可视化库**

---

### 8.14 单条会话的操作（次级菜单）

用户对单条会话可执行：

- **置顶 / 取消置顶**
- **分享会话深链**（生成可发给同事的 URL）
- **在 Finder 中显示**（打开 session 文件所在目录，仅 macOS 桌面）
- **复制 session id**
- **复制工作目录路径**
- **导出会话**（可选：导出为 Markdown / JSON）
- **打开工作目录**（在编辑器中，可选）

触发方式应支持：**右键**（桌面）+ **长按**（触屏）+ **菜单按钮**（hover 时浮现）。

---

### 8.15 单条事件的操作

时间线上每条事件，用户可：

- **查看原始 JSON**
- **复制内容**（如复制助手回复、复制命令 stdout、复制 diff 等）
- 特定事件可有专属操作：
  - 代码块的"**复制代码**"
  - 图像的"**保存图像**"
  - 链接的"**打开**"
  - 文件 diff 的"**在编辑器中打开该文件**"

---

### 8.16 启动新任务

用户可从应用内启动一条全新的 AI 会话。需要：

- **选择 AI 来源**：Claude / Codex
- **选择工作目录** —— 需要一个**目录浏览器**：
  - 可直接输入路径
  - 可面包屑导航（点击中间任意一级返回）
  - 可看到当前目录的**子目录列表**，点击进入
  - 可看到"**最近使用过的目录**"作为快捷入口
- **输入 prompt**（多行文本）
- **启动**按钮
- 启动后该会话应立即出现在会话列表中
- 启动前应可**取消**
- 启动后短时间内应可**终止**（杀掉后台进程）

---

### 8.17 分享

用户可分享：

#### 8.17.1 整个 App 的 LAN 链接
- 形如 `http://192.168.x.x:7800/?t=<token>`
- 含 token 安全保护（同一 Wi-Fi 才能访问，未授权打不开）
- 应显示**所有可达的本地 IP**（多网卡场景）
- 应可一键**复制 URL** 到剪贴板
- 应清晰提示**安全状况**（token 有效性）

#### 8.17.2 单条会话深链
- 形如 `http://192.168.x.x:7800/?t=<token>&type=claude&session=<sid>`
- 对方打开后直接定位到该会话

---

### 8.18 实时监听指示

某处应让用户感知到 App 是"活的"：

- 默认状态：一个"**实时监听中**"的指示（呼吸 / 微光 / 计数器都可）
- 后端**断开连接**：状态变化，提示用户
- 后端**正在接收新数据**：应有更明显的"活动"提示（但不打扰）

---

### 8.19 设置

需要提供可配置的设置项。**预期设置项很多**，需分模块组织：

#### 8.19.1 扫描配置
- Codex 会话目录（默认 `~/.codex/sessions`，可改）
- Claude 会话目录（默认 `~/.claude/projects`，可改）

#### 8.19.2 网络配置
- 监听端口（默认 7800）
- 绑定地址（默认所有网卡）
- token 模式：自动每次随机 / 固定指定 / 不需要

#### 8.19.3 外观
- 主题：暗 / 亮 / 跟随系统（**本期重点做暗色**）
- 字号缩放（可选）
- 是否显示原始 session id（可选）

#### 8.19.4 阅读偏好
- 默认事件类型过滤（开启哪几类）
- 思考块默认展开 / 折叠
- 工具调用默认展开 / 折叠
- 长内容自动截断的阈值

#### 8.19.5 通知（可选）
- 新事件到达时是否提示
- 新会话开始时是否提示

#### 8.19.6 自动更新
- 开 / 关
- 检查频率
- 是否自动下载

#### 8.19.7 关于
- 当前版本号
- 检查更新按钮
- 有可用更新时：**下载** + **应用并重启**
- 链接到项目主页 / 反馈渠道
- 致谢 / 许可证

设置项的控件类型可能包括：**开关、单行输入、多行输入、下拉、按钮、滑块、路径选择、颜色选择**。

设置应能**保存到本地配置文件** —— 重启后保留。
**新设置项的加入应不破坏既有布局**（设计要可扩展）。

---

### 8.20 自动更新

应用后台每 30 分钟拉取 GitHub Releases 检查新版本：

- 发现新版本时**不打扰用户阅读**
- 但需在某处有一个**"有更新可用"** 的低调提示（数字标记 / 小点 / 等）
- 用户点击后，应能看到**版本说明**（更新日志）
- 用户可触发**下载**（下载过程中显示进度）
- 下载完成后，应有**"应用更新（重启）"** 按钮
- 整个过程不应粗暴打断当前阅读

---

### 8.21 错误与异常

需要优雅处理以下情况：

- **后端断开**：状态变化，自动重连，重连成功后状态恢复
- **目录不存在**：友好提示，引导用户改设置
- **目录为空**：空状态提示（"还没有会话"），引导用户启动一个新任务
- **筛选无结果**：空状态提示（"没有匹配的会话"），可一键清空筛选
- **未选中任何会话**：友好空状态（**不是空白屏幕**）
- **会话不存在 / 已删除**：友好提示，自动跳回列表
- **网络断开**：不阻塞阅读已加载的会话
- **更新失败**：提示原因，可重试
- **发送消息失败**：提示原因，**输入框内容应保留**（不能让用户重打）

**所有错误都不能用浏览器原生 alert / 系统弹窗** —— 一律用应用内的轻量提示。
错误不应抖动 / 闪烁 / 红光大作 —— 用淡入的、可关闭的提示条。

---

### 8.22 会话深链入口

应用通过 URL 参数 `?type=&session=` 启动时：

- 应**自动定位**到指定会话并打开
- 如果会话不存在，给出友好提示并回到默认状态
- 如果 token 验证失败，给出鉴权提示

---

### 8.23 桌面外壳的特殊性（仅 macOS 原生 App）

在原生 App 形态下：

- 红绿灯三键由系统绘制 —— 左上角必须**留出空间**（约 116px 宽）
- 整个标题栏区域应**可拖拽窗口**
- 输入框、按钮、链接等可点击元素**不应可拖拽**（避免误触）
- 窗口圆角由系统提供
- **焦点 / 失焦时**窗口外观可有微妙差异（macOS 习惯）
- 窗口最小尺寸应合理（不能小到布局崩溃）

---

### 8.24 移动端（通过 LAN 链接在 iPhone / iPad 上访问）

- 屏幕窄了，布局必须**重新组织**（不是简单缩放）
- 触控替代鼠标 —— **hover 不可用**，需要等效的触控反馈
- **长按**应可触发等同"右键菜单"的功能
- 字号整体放大一档（手机阅读距离不同）
- 输入区需考虑**虚拟键盘弹起**后的视图压缩（safe-area / dvh）
- 不应在横向出现水平滚动条
- **视觉语言保持一致** —— 同一个 App 的两种形态，**不是两个 App**
- 桌面专属功能（如"在 iTerm2 中继续"）在移动端应隐藏

---

### 8.25 性能预算

不是直接的"功能"，但影响设计决策：

- 单会话 5000+ 事件，整体滚动应保持 60fps
- 首屏加载到能看清第一屏内容应在 300ms 内
- 新事件到达到呈现在屏幕上应在 100ms 内
- **不可以**给长列表每条都加复杂入场动画
- **不可以**对每条事件都做实时 highlight 计算
- 应在心里预演："5000 条事件在屏幕上滚动，会卡吗？"

---

## 9. 技术约束（你的设计要在这些条件下可实现）

- 运行在 macOS 的 WKWebView 中（Safari 内核）
- 前端是单个 HTML 文件 + Tailwind CSS + 原生 JS
- 不能引外网资源 —— 所有资源必须 inline / base64
- 不能依赖大型动画库 —— 优先 CSS 动画 / Web Animations API
- 字体使用系统字体：SF Pro Text、PingFang SC、SF Mono

- 红绿灯三键由系统绘制，左上角必须为它们留出空间
- 不能调用 macOS 原生组件 —— 所有 UI 是 Web 实现
- 支持暗 / 亮双模式（**本期重点做暗色**）

## 11. 感官验收标准（不是清单，是感觉）

我会用这些"感觉测试"来判断设计是否达标：

1. **三秒测试**：陌生人看一眼，能说出"这是 2027 年的 Apple 风格"
2. **凝视测试**：盯着空界面 30 秒，会感到平静而非无聊
3. **余光测试**：放在第二屏幕，余光能感觉到它在"呼吸"
4. **并排测试**：和 Linear / Things 3 / Notes 并排打开，气质同一家族
5. **呼吸测试**：每一处都有让眼睛休息的距离，没有挤迫感
6. **手指测试**：每一个可点击的地方，碰一下都有回应
7. **保持沉默测试**：界面在没事发生时**完全安静**
8. **细节测试**：放大到 200% 看任何一个角落，没有粗糙的细节
9. **三个月测试**：用户用了三个月仍然不舍得换皮肤
